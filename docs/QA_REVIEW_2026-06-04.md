## 🔴 High (우선 처리 권장)

### ⚠️ "조용한 데이터 손실" 클러스터 (H-2 ~ H-5)

### H-3. Ollama Cloud 추출 호출에 재시도/백오프 부재 (버그)
- **위치**: `lib/ollama_extractor.py:226-247`
- **무엇이 문제**: 일시적 클라우드 오류(429 과다요청 / 5xx 서버오류) 한 번에 문서 일부(최대 20페이지)가 영구 누락됩니다. `process_file`은 부분 결과라도 `{stem}.md`를 저장하므로, 재실행 시 "이미 존재"로 스킵되어 손실이 **고착**됩니다.
- **왜 문제(쉬운 말)**: 네트워크가 잠깐 흔들리면 그 페이지들이 영원히 사라지고, 다시 돌려도 복구되지 않습니다.
- **고치는 법**: `_ollama_vision`에 멱등 재시도(429/500/502/503/504 및 `httpx.HTTPError`에 대해 최대 3~5회, 지수 백오프 + jitter, `Retry-After` 헤더 존중) 추가. 추출은 읽기 전용이라 재시도 안전.

### H-4. 모델 응답 잘림(`finish_reason=length`) 미감지 (버그)
- **위치**: `lib/ollama_extractor.py:236-247`
- **무엇이 문제**: 응답이 토큰 한도로 잘려도 이를 감지하지 않고 **완전한 추출본으로 채택**합니다. AUTO 모드의 vision QA sanity_gate는 "길이 50% 미만"만 걸러서, 절반 이상 잘린 경우는 통과시킵니다.
- **왜 문제(쉬운 말)**: 표·핀맵이 중간에 잘린 채 저장되어 **데이터 무결성이 깨집니다**(에러 없이).
- **고치는 법**: `finish_reason`을 읽어 `'length'`면 경고 + (a) 더 작은 페이지 단위 재시도, (b) 잘림 마커 삽입, 또는 최소한 호출부에 `truncated=True` 전파.

### H-5. 추출 실패 시에도 부분 `{stem}.md` 기록 + 재실행 스킵 (버그)
- **위치**: `extract_all_via_pdf.py:574-577, 626-633`
- **무엇이 문제**: 일부 청크/페이지가 실패해도 부분 결과가 완성본처럼 저장되고, 존재 스킵 가드 때문에 재실행으로도 복구 불가.
- **왜 문제(쉬운 말)**: 불완전한 추출본이 "완성"으로 위장해 RAG 인덱싱에 그대로 흘러 들어갑니다.
- **고치는 법**: (1) 실패 청크가 있으면 `{stem}.partial.md`로 쓰거나 누락 마커(`<!-- MISSING pages X-Y: extraction failed -->`) 삽입. (2) 스킵 가드를 "실패 마커 없음 + 완전성 검증"으로 강화하거나 페이지별 manifest(상태 파일)로 누락 페이지만 보강. `reextract_figures.py` 패턴 일반화 가능.

### H-2. 청크 모드 net_tracer 교차검사가 잘못된 `[spurious?]` 플래그 생성 (버그)
- **위치**: `extract_all_via_pdf.py:192-231` (`apply_netcheck`) — `lib/net_crosscheck.py:386-442`와 상호작용
- **무엇이 문제**: `VISION_QA_NETCHECK=1` + 기본 청크 파이프라인(`CHUNK_SIZE=20`)에서, 멀티페이지 청크 MD 전체를 **단일 페이지 넷리스트**와 대조합니다. 그 결과 청크의 2~N 페이지 정상 부품 대부분이 `[spurious?]`로 오표기됩니다. (참고: `[spurious?]`+`[vector-confirmed]` 동시 부착은 mixed-line 억제 로직(`net_crosscheck.py:424-426`)으로 대부분 억제되나, 핵심 문제인 **타 페이지 부품 오표기**는 그대로 발생합니다.)
- **왜 문제(쉬운 말)**: 회로 검증 보조 표시가 거짓 경고를 남발해 결과를 오히려 오염시킵니다.
- **고치는 법**: 멀티페이지 청크를 단일 페이지 넷리스트와 대조하지 말 것 — (a) 청크 MD를 페이지별로 나눠 각 페이지 넷리스트와 대조, (b) 청크 페이지 합집합 넷리스트 1개 구성 후 대조, 또는 (c) `start_page==end_page`일 때만 `apply_netcheck` 실행.

### H-6. 앙상블이 동일 페이지 PNG를 N회 중복 렌더링 (성능)
- **위치**: `lib/vision_qa_ensemble.py:721-723`
- **무엇이 문제**: 앙상블 검증은 LLM 호출만 N번 독립이면 되는데, **렌더된 이미지는 바이트 동일**인데도 N번 다시 래스터화합니다. 300 DPI 고밀도 회로도 래스터화는 무겁습니다.
- **고치는 법**: `review_ensemble`에서 PNG를 한 번만 렌더해 미리 만든 경로를 LLM 검증 호출 N회에 전달. `_review_via_claude_cli`가 이미 렌더된 `png_paths`를 받도록 리팩터링.

- **M-7 (extract_all_via_pdf.py:204-211)** 청크 경로에서 페이지마다 net_tracer 서브프로세스 재실행(매번 PDF 재파싱) → `run_net_tracer.py`가 페이지 범위를 받아 단일 호출로 페이지별 JSON 출력.
- **M-8 (extract_all_via_pdf.py:614-625)** 어떤 실패든 멀티페이지 청크 전체를 처음부터 재추출(페이지별 resume 없음) → 청크 실패 시 페이지/소청크 단위 재추출로 폴백 + 제한 재시도.
